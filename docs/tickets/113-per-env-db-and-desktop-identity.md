# 113 — Per-environment Postgres DB + distinct desktop identity & launch scripts

- **Type:** Implementation
- **Status:** Done

## Goal
Build on [112](112-three-environments.md) so the three environments are fully
isolated and can run side by side locally:

1. **One separate Postgres database per environment** — dev/staging/production never
   share state.
2. **Distinct desktop app identity per environment** — different bundle identifier so
   all three install side by side, plus a name prefix + ribbon icon so dev/staging are
   obviously not production at a glance.
3. **Scripts** to start each environment locally on its own DB.

## 1. Separate Postgres database per env
| Env | Database | Backend port | Config template |
|-----|----------|--------------|-----------------|
| dev/local | `ctracking` | 8080 | `apps/backend/.env.example` |
| staging | `ctracking_staging` | 8081 | `apps/backend/.env.staging.example` |
| production | `ctracking_prod` | 8082 | `apps/backend/.env.prod.example` |

- `scripts/dev-db.sh` now creates the per-env databases (`ctracking_staging`,
  `ctracking_prod`) alongside the default `ctracking`, idempotently, in the one
  `ctracking-dev-db` container.
- New `.env.staging.example` / `.env.prod.example` templates carry a distinct
  `DATABASE_URL`, `PORT`, `STORAGE_DIR`, and `JWT_SECRET` per env. Real per-env files
  (`.env.staging`, `.env.prod`) and the `storage-staging/` `storage-prod/` blob dirs
  are gitignored.
- The backend reads config via godotenv, which does **not** override the process
  environment — so `start-env.sh` simply exports the env's vars and never clobbers a
  developer's committed `.env`.

## 2. Desktop identity per env
Bundle identifiers (so all three coexist on one machine):

| Env | `productName` | Bundle identifier | Window title | Icon |
|-----|---------------|-------------------|--------------|------|
| dev | BiBoTracking Dev | `com.briannguyen.bibotrackingdev` | `BiBoTracking [DEV]` | orange "DEV" ribbon |
| staging | BiBoTracking Staging | `com.briannguyen.ctracking` (unchanged) | `BiBoTracking [STAGING]` | purple "STG" ribbon |
| production | employeetrack | `com.briannguyen.bibotracking` | BiBoTracking | plain |

- Base `tauri.conf.json` is the **production** config; its identifier moved from
  `com.briannguyen.ctracking` → `com.briannguyen.bibotracking`. `productName` stays
  `employeetrack` so the `build-desktop-dmg` skill's artifact paths are unchanged.
- Dev/staging are deep-merge overrides applied with `tauri --config`:
  `src-tauri/tauri.dev.conf.json`, `src-tauri/tauri.staging.conf.json`. They override
  `productName`, `identifier`, the window title, and `bundle.icon` only — base
  `bundle.active/targets/macOS` survive the merge.
- Ribbon icons are generated by `scripts/gen-env-icons.py` (Pillow + `iconutil`) from
  the production `icons/icon.png` into `icons/dev/` and `icons/staging/` (PNGs + `.ico`
  + `.icns`). Re-run it if the base icon changes.

## 3. Launch scripts
- `scripts/start-env.sh <dev|staging|production> [backend|desktop|all]` — brings the
  DB up, starts the backend for that env (its own DB + port), and runs the matching
  desktop build (`--no-default-features --features <feature>` + the env's tauri
  config), with `CTRACKING_BACKEND_URL` pointed at the local backend port. `all`
  (default) runs backend in the background and the desktop in the foreground, tearing
  the backend down on exit.
- `scripts/dev-db.sh` extended to provision the per-env databases.
